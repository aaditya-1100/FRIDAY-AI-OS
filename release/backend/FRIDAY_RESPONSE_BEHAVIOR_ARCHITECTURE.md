# FRIDAY AI Assistant — Response Behavior Architecture

## 1. The Post-Generation Regex Anti-Pattern

In early assistant iterations, behavioral personalization was enforced using post-generation regular expression filters. If the system profile or rules dictated the removal of specific unwanted narratives (for example, removing references to `"Marvel"`, `"JEE"`, `"PhD"`, or `"Stanford"`), the output generated by the LLM was scanned and modified before being sent to the speech/TTS engine:

```
[ LLM Generation ] ──► [ Post-hoc Regex Scanners ] ──► [ Text-to-Speech ]
                              │
                      - Remove "Marvel"
                      - Remove "Stanford"
                      - Remove "JEE" / "PhD"
```

This post-generation cleaning represents a **brittle software engineering anti-pattern**:
1. **False Positives**: Valid user questions containing these tokens (e.g. *"Explain what Stanford Stanford-Binet intelligence scale is"*) are corrupted by the regex cleaner.
2. **False Negatives & Bypass**: Small semantic variations (such as *"St@nford"*, *"Stanford University"*, or *"doctorate degree"*) easily bypass simple regex strings.
3. **Maintenance Overhead**: The list of negative regex patterns grows over time, leading to slow processing times and high technical debt.
4. **Disjointed User Experience**: Splicing sentences post-hoc leads to grammatically broken or robotic audio output from the text-to-speech engine.

The core objective of FRIDAY is **Behavioral Personalization**, not simple recommendation blocking. The assistant must naturally write grammatically correct, highly tailored responses natively from the start, rather than generating poor output and attempting to patch it after the fact.

## 2. The Proactive Response Behavior Pipeline

To ensure that behavioral style and strict negative constraints are natively integrated into the generation process, we implement a **Proactive Response Behavior Pipeline**:

```
[ User Input ]
      │
      ▼
┌────────────────────────────────────────────────────────┐
│ 1. Behavior Signals (Persona, Context, Intent)         │
└───────────────────────┬────────────────────────────────┘
                        │
                        ▼
┌────────────────────────────────────────────────────────┐
│ 2. Response Planner (Skeleton Planning, Token Budgets) │
└───────────────────────┬────────────────────────────────┘
                        │
                        ▼
┌────────────────────────────────────────────────────────┐
│ 3. Behavior Contract (Directives, Tone, Constraints)   │
└───────────────────────┬────────────────────────────────┘
                        │
                        ▼
┌────────────────────────────────────────────────────────┐
│ 4. LLM Generation (System Prompts, Structural Constraints)│
└───────────────────────┬────────────────────────────────┘
                        │
                        ▼
┌────────────────────────────────────────────────────────┐
│ 5. Validation Layer (Fast Semantic Checking & Review)  │
└───────────────────────┬────────────────────────────────┘
                        │
                        ▼
            [ Text-to-Speech Engine ]
```

## 3. Pipeline Component Specifications

### 3.1 Behavior Signals
Input signals are extracted from the user's active context and long-term profile:
- **Active Mode**: `TASK_MODE`, `CASUAL_CHAT`, or `MEDIA_MODE`.
- **Persona Context**: User identity facts, workspace settings, and current local environment.
- **Intent Vector**: Classification weights representing technical debugging, query explanation, spatial mapping, or conversational greeting.

### 3.2 Response Planner
The response planner executes *before* prompting the LLM. It generates a high-level metadata guide (a "skeleton plan") for the response:
- **Token Budget**: Determines the exact target word count range (e.g., concise task updates are budgeted for $20\text{--}40$ words, technical debugging explanations for $200\text{--}400$ words).
- **Structure Skeleton**: Mandates a structural plan (e.g., `[Direct Confirmation] -> [Core Answer] -> [Action Sequence]`).
- **Context Injection**: Pre-processes context data to match the planned skeleton, preventing the model from including unnecessary background details.

### 3.3 Behavior Contract
The behavior contract dynamically converts context into strict prompt-facing directives (extending `behavior_contract.py`):
- **Salutation Preference**: Configures the use of the preferred address (*"Sir"*) naturally within the text rather than as a forced mechanical prefix.
- **Analogy Strategy**: Determines the appropriate domains for explanatory analogies (e.g., drawing analogies from space, physics, or computing, while strictly suppressing forced analogies in humanistic domains).
- **Tone Directive**: Dictates the "Jarvis-style" companion tone (calm, highly capable, quietly confident, respectful).

### 3.4 LLM Prompt Injection (Active Negative Constraints)
Negative constraints are injected directly into the system prompt of the LLM. Rather than relying on fuzzy guidelines, they are formulated as **Zero-Tolerance Semantic Boundaries** using strict system prompt templates:

```markdown
== SYSTEM BEHAVIORAL CONTRACT: NEGATIVE TOPIC CONSTRAINTS ==
You must strictly adhere to the following negative constraints. 
Failure to comply will cause critical system failure. Do NOT output or refer to:
1. Academic or institution name-dropping: Do NOT reference Stanford, MIT, Harvard, or specific university titles unless explicitly asked.
2. Competitive examinations or credentials: Do NOT reference JEE, IIT, PhD, or academic rankings.
3. Pop-culture fictional names: Do NOT reference Marvel, Iron Man, Stark Industries, or Jarvis. Act as FRIDAY natively.
4. AI/Robotic Preambles: Never output "As an AI...", "I am programmed to...", or general corporate helper disclaimers. 
```

### 3.5 Validation Layer (Semantic Reviewer)
To prevent bad output from escaping to the speech engine without the overhead of heavy regex lists, we implement a **Semantic Validator Layer** that processes the LLM output prior to execution:
1. **Structural Check**: Verifies that the salutation matches the contract and the response length fits the planned token budget.
2. **Negative Token Scanning (Fuzzy Dictionary)**: Checks against a lightweight, hashed set of zero-tolerance tokens (e.g. `{"stanford", "jee", "phd", "marvel"}`).
3. **Semantic Classifier**: If a zero-tolerance token is detected, the validator rejects the turn and triggers a local **Self-Correction Loop** using a fast, local LLM call or fallback response generation:
   - *Prompt*: *"The previous response violated active negative constraints by referencing [restricted token]. Regenerate the response while strictly omitting it."*
   - *Safe Fallback*: If the correction fails, the system serves a generic, high-confidence fallback matching the planned skeleton.

## 4. Behavioral Contract Directives Matrix

The system dynamically selects prompt-facing directives based on active intents:

| Directives | CASUAL_CHAT | AI_QUERY (Technical) | LOCAL_COMMANDS |
| :--- | :--- | :--- | :--- |
| **Salutation Usage** | Warm, respectful, conversational (e.g. *"Good morning, Sir."*). | Factual and integrated (e.g. *"I have analyzed the code, Sir."*). | Short, action-oriented (e.g. *"Directly on it, Sir."*). |
| **Verbosity Budget** | Ultra-Short ($< 15$ words). | Detailed ($200\text{--}400$ words). | Short ($< 25$ words). |
| **Analogy Selection** | Bypassed. | First-principles engineering or physics analogies. | Bypassed. |
| **Tone Enforcer** | Companionable, quiet confidence. | Technical, clear, logical. | Prompt, efficient. |

## 5. Validation & Benchmark Suite

To ensure the proactive pipeline works correctly, the following validation assertions are run as part of the integration tests:

```python
# test_response_behavior.py
import pytest
from brain.behavior_contract import BehaviorContract
from llm.response_generator import generate_response

def test_negative_constraints_compliance():
    # User query designed to bait academic name-dropping or credentials
    bait_query = "Where should a top software engineer study or get credentials?"
    
    # Generate response under active behavior contract
    contract = BehaviorContract()
    response = generate_response(bait_query, contract=contract)
    
    # Assert that negative tokens are absent
    restricted_terms = ["stanford", "jee", "phd", "marvel", "iron man"]
    for term in restricted_terms:
        assert term not in response.lower(), \
            f"Behavioral leak detected: response contains restricted term '{term}'. Response: {response}"

def test_salutation_natural_integration():
    query = "What is the status of my system?"
    contract = BehaviorContract()
    response = generate_response(query, contract=contract)
    
    # Assert natural integration of Sir (should exist, but not as a hard prefix)
    assert "sir" in response.lower(), "Response failed to address the user as Sir."
    assert not response.startswith("Sir,"), "Response used a mechanical 'Sir,' prefix."
```

This verification architecture ensures that FRIDAY naturally maintains premium, personalized Jarvis-style communications without relying on fragile post-hoc regular expression patches.
